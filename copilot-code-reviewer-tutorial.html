Copilot Code Reviewer 深度实现教程

深入理解 GitHub Copilot SDK 驱动的 GitLab MR 自动代码审查系统

项目概览

copilot-code-reviewer 是一个基于 GitHub Copilot SDK 的自动化代码审查工具。它监听 GitLab Merge Request 的 Webhook 事件，自动克隆代码仓库、生成 diff，调用 Copilot SDK 进行 AI 分析，然后将审查结果以 summary note 和 inline discussion 的形式回写到 GitLab MR。

核心价值

这个项目展示了如何将 GitHub Copilot SDK（原本为 VS Code 扩展设计的 AI 能力）独立于编辑器之外，构建成一个服务端 AI Agent。它不是简单的 API 调用，而是利用了 Copilot SDK 的完整 Agent 能力：会话管理、流式输出、权限控制、自定义工具等。

系统架构

GitLab MR → Webhook → FastAPI 后端 → Copilot SDK
AI 分析结果 → 解析 & 过滤 → GitLab API → MR Note/Discussion
React 前端 ↔ FastAPI 后端 ↔ JSON 文件存储

系统分为三个主要层次：

1触发层：GitLab Webhook 在 MR 创建/更新时触发通知
2处理层：FastAPI 后端接收 Webhook，克隆仓库、生成 diff、调用 Copilot SDK 分析、解析结果
3反馈层：将审查结果通过 GitLab API 写回为 MR Note 和 Inline Discussion

完整数据流

一次完整的代码审查流程如下：

1Webhook 触发：开发者在 MR 评论中包含触发关键词（如 /copilot-review），GitLab 发送 Note 事件到后端
2验证 & 去重：service.verify_webhook_token() 验证签名，maybe_schedule_review() 检查事件类型、关键词、去重
3后台任务：通过 FastAPI BackgroundTasks 异步执行 _run_review_job()
4获取 MR 数据：调用 GitLab API 获取 MR 详情和 changes
5克隆仓库：reviewer.clone_or_update_repo() 使用 GitPython 克隆/更新源和目标仓库
6生成 Diff：reviewer.build_diff_from_repo() 生成 unified diff
7构建 Prompt：reviewer._build_prompt() 将 diff + MR 元数据组装成结构化 prompt
8调用 Copilot SDK：reviewer.review() 创建 CopilotClient → 创建 Session → send_and_wait
9流式接收：通过 session 事件监听器实时捕获 assistant.message_delta 流式文本
10解析结果：reviewer._parse_structured_review() 将 AI 输出解析为 StructuredReview 模型
11过滤 & 发布：按严重度过滤 findings，发布 summary note 和 inline discussions

Copilot SDK 深度解析

这是本教程最核心的部分。Copilot SDK（copilot Python 包）是 GitHub Copilot 的官方 SDK，它提供了完整的 AI Agent 能力。项目中的 copilot-sdk-quick-start/app.py 是一个绝佳的学习入口，它展示了 SDK 的所有参数和能力。

理解 Copilot SDK 的关键

Copilot SDK 不是简单的 LLM API 封装。它是一个完整的 Agent 框架，包含：客户端生命周期管理、会话（Session）管理、权限控制、工具注册、流式输出、MCP 服务器、自定义 Agent 等能力。理解它的最好方式是从 Quick Start 示例开始。

CopilotClient 生命周期

SDK 的使用始于 CopilotClient，它管理着与 Copilot 服务的连接：

from copilot import CopilotClient, PermissionHandler, define_tool

# 1. 创建客户端实例
client = CopilotClient()

# 2. 启动客户端（建立连接）
await client.start()

# ... 使用客户端创建会话、发送消息 ...

# 3. 停止客户端（释放资源）
await client.stop()

CopilotClient 的生命周期非常严格：

✅ 正确用法

client = CopilotClient()
await client.start()
try:
    # 创建会话、发送消息
    ...
finally:
    await client.stop()

❌ 错误用法

client = CopilotClient()
# 忘记 start() 就直接使用
session = await client.create_session(...)
# 忘记 stop() 导致资源泄漏

在实际项目中（reviewer.py），客户端还可以传入配置：

client = CopilotClient({
    "cwd": context.repo_path,     # 工作目录，影响文件访问权限
    "log_level": "error"        # 日志级别
})
cwd 参数的作用

cwd 参数非常重要！它决定了 Copilot Agent 的工作目录。SDK 会基于此目录来解析文件路径、执行命令等。在 reviewer 中，它被设置为克隆下来的仓库路径，这样 AI 就能"看到"项目代码。

create_session 参数全解

create_session() 是 SDK 最核心的方法，它创建一个 AI 会话。Quick Start 示例详细展示了所有可用参数：

session_config: dict[str, Any] = {

send_and_wait 机制

创建会话后，通过 send_and_wait() 发送消息并等待 AI 响应：

message_options: dict[str, Any] = {
    # ===== 必需参数 =====
    "prompt": "Call the multiply_numbers tool to calculate 6 * 7...",
    # 发送给会话的消息文本。
    # 这是 AI 处理的主要输入。

    # ===== 默认行为参数 =====
    "attachments": None,
    # 文件、目录或选区附件。
    # 可以附加文件内容供 AI 参考。

    "mode": None,
    # 发送模式。None 使用 SDK 默认（enqueue）。
    # 可选 "immediate" 跳过队列立即处理。
}

# 超时设置（秒）
send_timeout = 60.0

response = await session.send_and_wait(
    message_options,
    timeout=send_timeout,
)

# 获取响应内容
content = response.data.content

send_and_wait 的工作流程：

发送 prompt → SDK 入队 → AI 处理 → 返回响应
send_and_wait vs 事件监听

send_and_wait 会阻塞直到 AI 完成响应（或超时）。但如果启用了 streaming=True，你还可以通过 session.on() 注册事件监听器来实时接收增量输出。这两种机制可以同时使用！

Streaming 流式输出

流式输出是本项目中 Copilot SDK 使用的关键特性。在 reviewer.py 中，会话配置了 streaming=True，然后通过事件监听器实时捕获 AI 输出：

# 1. 配置会话启用流式
session_params = {
    "model": settings.copilot_model,
    "on_permission_request": PermissionHandler.approve_all,
    "streaming": True,    # ← 启用流式输出
}

# 2. 注册事件处理器
def handle_event(event) -> None:
    event_type = getattr(getattr(event, "type", None), "value",
                         getattr(event, "type", "unknown"))
    data = getattr(event, "data", None)
    delta_text = self._extract_delta_text(data)

    # 3. 处理流式文本增量
    if event_type == "assistant.message_delta" and delta_text:
        append_stream_text("copilot-stream", delta_text)
        return

    # 4. 忽略推理和流式控制的增量
    if event_type in {"assistant.reasoning_delta",
                      "assistant.streaming_delta"}:
        return

session.on(handle_event)

# 5. 发送消息
response = await session.send_and_wait(
    {"prompt": prompt},
    timeout=settings.copilot_timeout_seconds
)

SDK 会发出以下类型的事件：

事件类型含义本项目处理
assistant.message_deltaAI 输出的文本增量✅ 捕获并累积到 stream_buffer
assistant.reasoning_deltaAI 推理过程的增量❌ 忽略
assistant.streaming_delta流式控制增量❌ 忽略
其他事件工具调用、状态变更等记录日志
为什么优先使用累积的流式输出？

项目中有这样的逻辑：if accumulated: content = accumulated。这是因为 response.data.content 在某些提供商配置下可能被截断（如 Ollama），而通过 message_delta 事件累积的文本是完整的原始模型输出。所以累积流式输出是比 response.content 更可靠的结果来源。

自定义工具系统

Copilot SDK 允许你注册自定义工具，AI 可以在会话中主动调用它们。Quick Start 示例展示了完整的工具定义流程：

from copilot import define_tool
from pydantic import BaseModel, Field

# 1. 定义工具的参数模型
class MultiplyParams(BaseModel):
    a: int = Field(description="The first integer to multiply")
    b: int = Field(description="The second integer to multiply")

# 2. 使用 @define_tool 装饰器注册工具
@define_tool(description="Multiply two integers and return the product")
def multiply_numbers(params: MultiplyParams) -> str:
    product = params.a * params.b
    return str(product)

# 3. 在创建会话时传入工具
session_config = {
    "tools": [multiply_numbers],  # ← 注册工具
    ...
}

工具系统的运作流程：

用户 prompt → AI 决策 → 调用工具 → 返回结果 → AI 继续推理
工具定义三要素

① 参数模型：Pydantic BaseModel，描述工具输入，Field 的 description 会告诉 AI 每个参数的含义
② @define_tool 装饰器：注册工具，description 告诉 AI 这个工具做什么
③ 实现函数：接收参数模型，返回字符串结果

在本项目中，reviewer 没有注册自定义工具，而是完全依赖 AI 的文本生成能力。但工具系统为扩展留下了空间——例如，你可以注册一个"查询项目编码规范"的工具，让 AI 在审查时参考。

Provider 配置

SDK 支持两种提供商模式：

默认 Copilot 提供商

使用 GitHub Copilot 的后端服务，需要有效的 Copilot 订阅。这是默认模式，无需额外配置。

session_params = {
    # 不设置 provider
    "model": "claude-sonnet-4",
}

BYOK 提供商

Bring Your Own Key——使用自己的 API Key 和端点。支持 OpenAI 兼容接口。

provider = {
    "type": "openai",
    "base_url": "https://api.openai.com/v1",
    "api_key": "sk-...",
}
session_params = {
    "provider": provider,
}

在 reviewer.py 中，provider 配置是动态的：

if settings.copilot_provider_type:
    provider = {"type": settings.copilot_provider_type}
    if settings.copilot_provider_base_url:
        provider["base_url"] = settings.copilot_provider_base_url
    if settings.copilot_provider_api_key:
        provider["api_key"] = settings.copilot_provider_api_key
    session_params["provider"] = provider

Quick Start 示例拆解

copilot-sdk-quick-start/app.py 是一个完整的 SDK 使用示例，让我们逐行拆解：

import asyncio
from copilot import CopilotClient, PermissionHandler, define_tool
from pydantic import BaseModel, Field
from typing import Any

# ========== 第一步：定义自定义工具 ==========
class MultiplyParams(BaseModel):
    a: int = Field(description="The first integer to multiply")
    b: int = Field(description="The second integer to multiply")

@define_tool(description="Multiply two integers and return the product")
def multiply_numbers(params: MultiplyParams) -> str:
    product = params.a * params.b
    print(f"[tool] multiply_numbers called with a={params.a}, b={params.b}")
    return str(product)

# ========== 第二步：主函数 ==========
async def main():
    # 创建并启动客户端
    client = CopilotClient()
    await client.start()

    try:
        # ========== 第三步：配置会话 ==========
        session_config: dict[str, Any] = {
            "on_permission_request": PermissionHandler.approve_all,
            "tools": [multiply_numbers],
            # ... 其他参数见上文详解 ...
        }

        # ========== 第四步：创建会话 ==========
        async with await client.create_session(session_config) as session:
            # ========== 第五步：发送消息 ==========
            message_options: dict[str, Any] = {
                "prompt": "Call the multiply_numbers tool to calculate 6 * 7, "
                          "then write a haiku about the result.",
            }

            send_timeout = 60.0
            response = await session.send_and_wait(
                message_options,
                timeout=send_timeout,
            )
            print(response.data.content)
    finally:
        await client.stop()

asyncio.run(main())
关键观察

注意 async with await client.create_session(...) 的用法——create_session 本身是异步的（需要 await），返回的是一个异步上下文管理器（需要 async with）。这确保了会话资源的正确获取和释放。

Reviewer 核心逻辑

reviewer.py 是整个项目最核心的文件，它封装了 Copilot SDK 的调用和结果处理。

Prompt 构建

_build_prompt() 方法构建了一个精心设计的结构化 prompt：

def _build_prompt(self, context: MergeRequestContext) -> str:
    files = []
    total = 0
    for item in context.changed_files:
        diff = (item.get('diff') or "")[:MAX_FILE_PATCH_CHARS]  # 限制单文件 diff 长度
        block = (
            f"File: {item.get('new_path')}\n"
            f"Old path: {item.get('old_path')}\n"
            f"New file: {item.get('new_file')}\n"
            f"Renamed: {item.get('renamed_file')}\n"
            f"Deleted: {item.get('deleted_file')}\n"
            f"Patch:\n{diff}"
        )
        if total + len(block) > MAX_TOTAL_DIFF_CHARS:  # 限制总 diff 长度
            break
        files.append(block)
        total += len(block)

    return textwrap.dedent(f""""
        You are an expert GitLab merge request reviewer.
        Write all natural-language output fields in {context.review_language}.
        ...

        Return ONLY valid JSON. No markdown fences. No commentary.
        Schema:
        {{
          "overall_assessment": "short paragraph",
          "findings": [
            {{
              "title": "short finding title",
              "severity": "high|medium|low",
              "file_path": "path/to/file or null",
              "line": 123,
              "details": "why it matters",
              "recommendation": "concise recommendation",
              "suggested_code": "optional replacement snippet...",
              "code_language": "optional language like python|javascript|typescript"
            }}
          ],
          "suggested_follow_ups": ["item 1", "item 2"]
        }}

        Important constraints:
        - Prefer findings anchored to changed lines in the new file.
        - Use file_path and line only when you are confident...
        - If there are no material issues, return an empty findings array.
        ...

        Merge Request metadata:
        - Project: {context.project_path}
        - MR IID: {context.mr_iid}
        - Title: {context.mr_title}
        - Author: {context.author}
        - Source branch: {context.source_branch}
        - Target branch: {context.target_branch}
        - URL: {context.web_url}
        - Trigger comment: {context.trigger_comment}

        Description:
        {context.mr_description or '(none)'}

        Unified diff:
        {unified_diff}

        Per-file patches:
        {per_file_patches}
        """).strip()
Prompt 设计要点

① 角色设定："You are an expert GitLab merge request reviewer"
② 输出约束：要求返回纯 JSON，不要 markdown 代码块
③ Schema 定义：明确定义 JSON 结构，包括每个字段的含义
④ 重要约束：优先关联到实际变更行、不确定时不标注行号、无问题则返回空数组
⑤ 上下文信息：MR 元数据 + unified diff + 逐文件 patch
⑥ 长度限制：单文件 diff ≤ 18000 字符，总 diff ≤ 60000 字符

会话与流式处理

review() 方法是 reviewer 的核心入口，它完整展示了如何在生产环境中使用 Copilot SDK：

async def review(self, context, log_callback=None,
           stream_start_callback=None, stream_chunk_callback=None,
           stream_end_callback=None) -> StructuredReview:
    prompt = self._build_prompt(context)

    # 配置会话参数
    session_params = {
        "model": settings.copilot_model,
        "on_permission_request": PermissionHandler.approve_all,
        "streaming": True,  # 启用流式输出
    }

    # 动态配置 provider
    if settings.copilot_provider_type:
        provider = {"type": settings.copilot_provider_type}
        ...
        session_params["provider"] = provider

    # 创建客户端
    client = CopilotClient({"cwd": context.repo_path, "log_level": "error"})
    await client.start()

    # 流式状态管理
    stream_state = {"open": False, "last_chunk": None, "source": None}
    stream_buffer = []

    try:
        async with await client.create_session(session_params) as session:
            # 定义流式文本处理函数
            def append_stream_text(source, text):
                # 去重：相同来源的相同文本不重复添加
                if text == stream_state["last_chunk"] and source == stream_state["source"]:
                    return
                # 来源变化时关闭前一个流
                if stream_state["open"] and stream_state["source"] != source:
                    close_stream_if_needed()
                # 新来源时开启流
                if not stream_state["open"]:
                    if stream_start_callback:
                        stream_start_callback(source)
                    stream_state["open"] = True
                    stream_state["source"] = source
                stream_buffer.append(text)
                if stream_chunk_callback:
                    stream_chunk_callback(text)
                stream_state["last_chunk"] = text

            # 注册事件处理器
            def handle_event(event):
                event_type = getattr(getattr(event, "type", None), "value",
                                     getattr(event, "type", "unknown"))
                data = getattr(event, "data", None)
                delta_text = self._extract_delta_text(data)

                if event_type == "assistant.message_delta" and delta_text:
                    append_stream_text("copilot-stream", delta_text)
                    return

                if event_type in {"assistant.reasoning_delta", "assistant.streaming_delta"}:
                    return

                # 非流式事件时关闭当前流
                if stream_state["open"] and event_type not in {...}:
                    close_stream_if_needed()

            session.on(handle_event)

            # 发送消息并等待
            response = await session.send_and_wait(
                {"prompt": prompt},
                timeout=settings.copilot_timeout_seconds
            )
            close_stream_if_needed()

            # 优先使用累积的流式输出
            content = getattr(response.data, "content", None)
            accumulated = "".join(stream_buffer).strip() if stream_buffer else None
            if accumulated:
                content = accumulated

            return self._parse_structured_review(content)
    finally:
        if stream_state["open"] and stream_end_callback:
            stream_end_callback()
        await client.stop()

结构化结果解析

AI 返回的 JSON 需要被解析为结构化的 StructuredReview 对象：

def _parse_structured_review(self, content: str) -> StructuredReview:
    text = content.strip()

    # 处理 markdown 代码块包裹的情况
    candidate = text
    if text.startswith("```"):
        match = JSON_BLOCK_RE.search(text)
        if match:
            candidate = match.group(1).strip()

    # 解析 JSON 并验证
    try:
        return StructuredReview.model_validate(json.loads(candidate))
    except (json.JSONDecodeError, ValidationError) as exc:
        raise RuntimeError(f"Unable to parse Copilot review JSON: {exc}")

这里有一个精妙的正则表达式处理：

JSON_BLOCK_RE = re.compile(r"`(?:json)?\s*(.*?)```", re.DOTALL)
为什么需要这个正则？

尽管 prompt 要求 AI 返回纯 JSON（"No markdown fences"），但 AI 有时仍会用 ```json ... ``` 包裹输出。正则只在文本以 ``` 开头时才尝试提取，避免错误匹配 JSON 字符串值中嵌入的代码块（如 suggested_code 字段）。

解析后，reviewer 还提供了丰富的渲染方法：

  • render_summary_markdown()：将审查结果渲染为 Markdown 格式的 summary note
  • render_inline_comment()：将单个 finding 渲染为 inline discussion 的内容
  • filter_findings_by_min_severity()：按严重度过滤 findings
  • resolve_position()：将 finding 的 file_path + line 解析为 GitLab discussion position
  • build_position_index()：从 diff 构建行号索引，用于定位 inline comment

Severity 枚举

Severity = Literal["high", "medium", "low"] — 使用 Python 的 Literal 类型确保严重度只能是这三个值之一。在 reviewer 中，严重度还有排序：{"low": 1, "medium": 2, "high": 3}，用于过滤。

        return self.model_cls.model_validate(payload)

    def save(self, state):
        self.path.write_text(

        )

SSE 日志流

日志流使用 Server-Sent Events (SSE) 实现实时推送：

async def event_generator():
    # 发送初始快照
    initial = job_log_store.read(job_id)
    yield f"event: snapshot\ndata: {json.dumps({'content': initial})}\n\n"

    while True:
        await asyncio.sleep(1)
        current = job_log_store.read(job_id)
        if current != last_sent:
            appended = current[len(last_sent):]
            yield f"event: append\ndata: {json.dumps({'content': appended})}\n\n"
            last_sent = current

        # 任务完成时发送状态事件
        latest_job = job_store.get_by_job_id(job_id)
        if latest_job.status in {"completed", "failed", "skipped"}:
            yield f"event: status\ndata: ..."
            break

return StreamingResponse(
    event_generator(),
    media_type="text/event-stream",
    headers={"Cache-Control": "no-cache", ...},
)

总结与扩展

项目核心设计模式

1SDK as Agent Framework：不是简单调用 LLM API，而是利用 Copilot SDK 的完整 Agent 能力（会话、流式、权限、工具）
2Structured Output：通过精心设计的 prompt 约束 AI 返回结构化 JSON，再用 Pydantic 验证
3Streaming First：优先使用流式累积输出而非 response.content，确保结果完整性
4Severity-based Filtering：通过严重度分级控制 inline comment 的发布，避免低价值评论噪音
5Local Git Diff：克隆仓库本地生成 diff，而非依赖 GitLab API 的 diff，获得更精确的行号定位

可扩展方向

基于 SDK 能力的扩展思路

自定义工具：注册"查询编码规范"、"运行 lint"等工具，让 AI 在审查时主动调用
MCP 服务器：接入 MCP 协议的外部工具/数据源
自定义 Agent：注册专门的 Security Agent、Performance Agent 等
多轮对话：利用 infinite_sessions 实现开发者与 AI 的多轮审查讨论
BYOK：接入自托管模型，降低成本或满足数据合规要求
